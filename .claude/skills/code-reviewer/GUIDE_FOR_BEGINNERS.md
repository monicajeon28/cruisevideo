## . 왜  가 필요한가

### 비유로 이해하기 "편집자"

**개발자  작가**
소설을 쓰는 데 집중합니다.

**   편집자**
- "이 문장이 너무 길어요" (복잡도 분석)
- "앞에서 같은 내용을 썼는데요" (중복 코드 탐지)
- "독자가 이해하기 어려워요" (가독성 점수)
- "전체적으로 + 입니다" (통합 점수)

### 실제 문제 셀프 체크의 한계


 셀프 체크가 안 되는 이유 

개발자가 직접 작성한 코드를 검토할 때
. "내가 쓴 거니까 당연히 이해됨" → 다른 사람은 못 알아봄
. "이게 최선이야" → 더 좋은 방법이 있음
. "일단 돌아가니까 " → 나중에 버그 폭탄
. "피곤해서 대충..." → 품질 저하

결과
- 다른 개발자 "이 코드 뭐야"
- 개월 후 본인 "내가 이걸 썼다고"


** 는 "객관적인 기준"으로 코드를 평가합니다.**

### .   (품질 관문)

** 란**
코드가 통과해야 하는 최소 기준점


  # 이것 못 통과하면 머지 불가
  - "  에러 없음"
  - "보안 취약점 없음"
  - "하드코딩 시크릿 없음"

  # 권고사항, 통과 못해도 머지 가능
  - "복잡도 ≤ "
  - "파일 길이 ≤ 줄"
  - "테스트 커버리지 ≥ %"


**비유 공항 보안 검색**

  금속 탐지기
  → 금속 반응 있으면 탑승 불가

  - 검색
  → 수상한 물건 있으면 추가 검사, 통과는 가능


### .   탐지

** 이란**
"냄새 나는 코드" - 당장 에러는 안 나지만 문제를 일으킬 코드


대표적인

  
    냄새 "함수가 줄이 넘음"
    문제 "이해하기 어렵고 테스트하기 힘듦"
    해결 "작은 함수로 분리"

  
    냄새 "클래스가 줄이 넘음"
    문제 "책임이 너무 많음 ( 위반)"
    해결 "역할별로 클래스 분리"

  
    냄새 "같은 코드가 여러 곳에 있음"
    문제 "수정 시 여러 곳 고쳐야 함"
    해결 "공통 함수로 추출"

  
    냄새 "실행되지 않는 코드가 있음"
    문제 "코드베이스 복잡도 증가"
    해결 "삭제"

  
    냄새 "다른 클래스 데이터를 더 많이 사용"
    문제 "잘못된 위치에 있는 로직"
    해결 "해당 클래스로 메서드 이동"

  
    냄새 "모든 걸 다 하는 클래스"
    문제 "수정 시 전체 시스템에 영향"
    해결 "책임 분리"


### . 복잡도 분석

**복잡도란**
코드가 얼마나 복잡한지 수치화한 것


   (순환 복잡도) 

분기점(, , ,  등) 개수를 세어 복잡도 측정

예시
┌─────────────────────────────────────┐
│  ()                │
│    (  )            // +      │
│      (  )         // +      │
│        ''                 │
│                               │
│        ''              │
│                                    │
│                               │
│      ''              │
│                                    │
│                                    │
│                                     │
│ 복잡도  (기본) + (분기)       │
└─────────────────────────────────────┘

복잡도 기준
  -    단순, 이해하기 쉬움
  -   보통, 주의 필요
  -  복잡, 리팩토링 권장
  +    매우 복잡, 분리 필수


**  (인지 복잡도)**

사람이 코드를 읽을 때 얼마나 힘든지 측정

추가 패널티
  - 중첩 깊이 ( 안에  안에 ...) +점씩
  - / 사용 +점
  - 재귀 호출 +점

인지 복잡도 기준
  -    쉬움
  -   보통
  +    어려움, 단순화 필요

  코드 품질 점  .  .

  총점     .점 → 등급

### . 등급별 피드백 예시


## 등급 (점)
 코드 품질이 매우 우수합니다.
-  원칙 준수
- 테스트 커버리지 %
- 보안 취약점 없음
→ 즉시 머지해도 됩니다!

---

## 등급 (점)
 전반적으로 좋으나 몇 가지 개선점이 있습니다.
- 권고 () 함수 복잡도가 입니다 (권장  이하)
- 권고 테스트 커버리지 % (권장 %)
→ 수정 후 머지하세요.

---

## 등급 (점)
 조건부 승인. 다음 사항을 확인하세요
- 필수 () 함수 줄 → 분리 필요
- 필수 중복 코드 곳 발견
- 권고 문서화 부족
→ 필수 사항 수정 후 다시 리뷰 요청하세요.

---

## 등급 (점)
 재작성이 필요합니다.
- 심각   취약점 발견
- 심각 하드코딩된 비밀번호
- 필수 테스트 없음
- 필수 줄짜리  
→ 보안 취약점부터 해결하세요!

## 네이밍
-   변수명이 의미를 담고 있는가
-   함수명이 동사로 시작하는가
-   클래스명이 명사인가
-   약어 사용을 피했는가

## 구조
-   함수가 줄 이하인가
-   들여쓰기가 단계 이하인가
-   파라미터가 개 이하인가

## 타입
-    타입 사용을 피했는가
-   / 처리가 있는가


### .   (표준 검토)


##   +

## 복잡도
-   순환 복잡도가  이하인가
-   인지 복잡도가  이하인가

## 중복
-   줄 이상 중복 코드가 없는가
-   복사-붙여넣기 코드가 없는가

## 의존성
-   순환 의존성이 없는가
-   의존성 방향이 올바른가

## 에러 처리
-   모든 에러 케이스를 처리하는가
-   적절한 에러 메시지가 있는가
