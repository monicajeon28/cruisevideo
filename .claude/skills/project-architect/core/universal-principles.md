## . 관심사 분리 (  )


원칙 "하나의 모듈은 하나의 책임만"

적용방법
  
    책임 "화면 렌더링, 사용자 입력 처리"
    금지 " 호출,  접근, 비즈니스 로직"
    
  
    책임 "비즈니스 규칙, 데이터 변환"
    금지 " 관심사, 직접  접근"
    
  
    책임 "데이터 저장/조회, 외부  통신"
    금지 "비즈니스 규칙,  로직"

검증질문
  - "이 파일을 수정하면 다른 레이어에 영향이 가는가"
  - "이 모듈의 책임을 한 문장으로 설명할 수 있는가"
  - "이 코드를 다른 프로젝트에 복사해도 작동하는가"

## . 단일 진입점 원칙 (  )


원칙 "모듈의  는 . 하나로 관리"

장점
  - 캡슐화 내부 구현 숨김
  - 리팩토링 용이 내부 변경이 외부에 영향 없음
  - 명확한  무엇을 사용할 수 있는지 명확

구현
  //. 
    // 외부에 노출할 것만 
         './/'
         './/'
         './/'
       ,    './'
    
    // 내부 구현은  하지 않음
    // -  (내부 컴포넌트)
    // -  (내부 유틸)
    // -  (내부 변환 함수)

규칙
   "     '//'"
   "     '////'"


## . 설정 외부화 원칙 ( )


원칙 "환경에 따라 달라지는 값은 코드에 넣지 않는다"

외부화대상
  필수
    -  엔드포인트 
    - 데이터베이스 연결 정보
    -  키, 시크릿
    - 기능 플래그
    
  권장
    - 페이지네이션 기본값
    - 타임아웃 시간
    - 재시도 횟수
    - 로깅 레벨

구현패턴
  /. 
        
       
         ..,
         (..)  ,
      ,
       
         ..,
      ,
       
         ..  '',
      ,
      
    
    // 필수값 검증
     () 
       (!..) 
          ('  ')
      
    


## . 실패 대비 설계 (  )


원칙 "모든 외부 연동은 실패할 수 있다고 가정"

대비항목
  호출
    - 타임아웃 설정
    - 재시도 로직
    - 폴백 처리
    - 에러 로깅
    
  데이터베이스
    - 연결 풀 관리
    - 트랜잭션 롤백
    - 데드락 방지
    
  외부서비스
    - 서킷 브레이커
    - 캐싱 (가능한 경우)
    - 그레이스풀 디그레이데이션

구현예시 
    ( ) 
     
        .()
      () 
      // 캐시에서 시도
          .($)
       ()  
      
      // 기본값 반환 ( )
         ,  ' ' 
    
  


## . 테스트 가능한 설계 ( )


원칙 "테스트하기 어려운 코드는 설계가 잘못된 것"

테스트용이성지표
  
    - 의존성 주입 사용
    - 순수 함수 많음
    - 작은 단위로 분리됨
    - 부작용이 명확히 분리됨
    
  
    - 전역 상태 의존
    - 하드코딩된 의존성
    - 거대한 함수/클래스
    - 숨겨진 부작용

개선패턴
   
    //  테스트 어려움 - 직접  호출
      ( ) 
          ('///' + )
      // ...
    
    
   
    //  테스트 용이 - 의존성 주입
      (
       ,
         
    ) 
          .()
      // ...
    
    
    // 테스트에서
         .() 
     ('', )
