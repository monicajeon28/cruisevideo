#     .

**코드 스멜 탐지기** - 문제가 작을 때 잡아서 산불 방지

## 핵심 철학



  원칙 "불씨가 산불 되기 전에 끄자"
  목표 "리팩토링이 필요한 시점을 조기에 알려줌"

  왜중요한가
    - 기술 부채 누적 방지
    - 유지보수 비용 절감
    - 버그 발생 가능성 감소
    - 코드 가독성 유지

  탐지시점
    - 코드 작성 직후 (즉시 피드백)
    - 코드 리뷰 시 ( 검토)
    - 리팩토링 계획 시 (부채 정리)

## 코드 스멜 분류

###   - 즉시 수정



  설명 "너무 많은 책임을 가진 클래스/모듈"
  탐지기준
    - 메서드 수  개
    - 프로퍼티 수  개
    - 파일 라인 수  줄
    - 의존성 수  개
  문제점
    - 수정 시 영향 범위 파악 불가
    - 테스트 어려움
    - 재사용 불가능
  해결책
    - 단일 책임 원칙() 적용
    - 관련 기능별로 클래스 분리
  예시
     
        
        () 
        () 
        () 
        () 
        () 
        () 
        // ... 개 더
      
     
         /* 사용자  */ 
         /* 이메일 발송 */ 
         /* 리포트 생성 */ 
         /* 결제 처리 */ 


  설명 "너무 긴 함수/메서드"
  탐지기준
    - 라인 수  줄
    - 들여쓰기 깊이  단계
    - 파라미터 수  개
    - 지역 변수  개
  문제점
    - 이해하기 어려움
    - 테스트 어려움
    - 재사용 불가능
  해결책
    - 의미 있는 단위로 함수 분리
    - 조기 반환( ) 적용
  예시
     
       () 
        // 줄의 로직...
         () 
           () 
             () 
              // 깊은 중첩...
            
          
        
      
     
       () 
        ()
        ()
        ()
        ()
        ()
      


  설명 "중복된 코드"
  탐지기준
    - 줄 이상 동일 코드 곳 이상
    - 유사한 로직 패턴 반복
  문제점
    - 수정 시 모든 곳 수정 필요
    - 일부만 수정하면 불일치 발생
  해결책
    - 공통 함수/모듈로 추출
    - 템플릿 메서드 패턴 적용


###   - 빠른 수정 권장



  설명 "다른 클래스의 데이터를 과도하게 사용"
  탐지기준
    - 다른 객체 메서드 회 이상 연속 호출
    - 다른 객체 프로퍼티 개 이상 접근
  문제점
    - 책임 분리 위반
    - 캡슐화 위반
  해결책
    - 메서드를 데이터가 있는 클래스로 이동
  예시
     
       () 
         . + . + . + .
      
     
        
        () 
           . + . + . + .
        
      


  설명 "항상 함께 다니는 데이터 그룹"
  탐지기준
    - 같은 파라미터 조합 곳 이상
    - 같은 필드 조합 반복
  문제점
    - 중복된 검증 로직
    - 변경 시 여러 곳 수정
  해결책
    - 데이터 클래스/인터페이스로 묶기
  예시
     
       (, , , ) 
       (, , , ) 
       (, , , ) 
     
         , , ,  
       ( ) 
       ( ) 


  설명 "기본 타입 남용"
  탐지기준
    - 문자열로 특수 값 표현 ( "")
    - 숫자로 코드 값 표현
    - 배열 인덱스로 의미 부여
  문제점
    - 타입 안전성 없음
    - 유효성 검증 반복
  해결책
    - 값 객체( ) 생성
    -  사용
  예시
     
         "" // 오타 가능
         "--" // 검증 안됨
     
         ,  
         ()  .()  


  설명 "파라미터가 너무 많은 함수"
  탐지기준
    - 파라미터  개
    -  파라미터 개 이상
  문제점
    - 호출 시 순서 혼동
    - 가독성 저하
  해결책
    - 파라미터 객체로 묶기
    - 빌더 패턴 사용
  예시
     
      (, , , , , , , )
     
      ( , , ,   ,  , ,  )


###   - 검토 권장



  설명 "사용되지 않는 코드"
  탐지기준
    - 호출되지 않는 함수
    - 도달 불가능한 코드
    - 주석 처리된 코드
  해결책 "삭제"


  설명 "미래를 위해 만든 사용 안 하는 추상화"
  탐지기준
    - 구현체가 개인 인터페이스
    - 사용 안 되는 파라미터
    - 오버엔지니어링된 패턴
  해결책 " - 필요할 때 만들기"


  설명 "코드를 설명하는 주석"
  탐지기준
    -  설명 주석 (코드로 알 수 있음)
    - 오래된 주석 (코드와 불일치)
  해결책
    - 코드 자체를 명확하게
    -  주석만 유지


  설명 "의미 없는 숫자/문자열 리터럴"
  탐지기준
    - 하드코딩된 숫자 (   )
    - 반복되는 문자열
  해결책 "상수로 추출"
  예시
     " (  )  ... "
     " (  .)  ... "


  설명 "깊게 중첩된 조건문"
  탐지기준
    -  중첩  단계
    -   체인  개
  해결책
    - 조기 반환 ( )
    - 전략 패턴
    - 다형성
  예시
     
       () 
         (.) 
           (.) 
            //  
          
        
      
     
       (!) 
       (!.) 
       (!.) 
      //

## 체크리스트


### 함수
  줄 이하인가
  파라미터 개 이하인가
  들여쓰기 단계 이하인가
  하나의 일만 하는가

### 클래스
  메서드 개 이하인가
  단일 책임인가
  의존성 개 이하인가

### 전체
  중복 코드가 없는가
  매직 넘버가 없는가
  죽은 코드가 없는가
