## . 왜  가 필요한가

### 비유로 이해하기 "건물의 경비 시스템"

**일반 개발자  집을 짓는 건축가**
집을 예쁘고 편리하게 만드는 데 집중합니다.

**   보안 전문가**
- "이 창문은 쉽게 깨질 수 있어요" (취약점 탐지)
- "현관 비밀번호가 예요" (약한 인증 경고)
- "열쇠를 우편함 아래 숨기셨군요" (하드코딩 시크릿 탐지)
- "방화문이 필요해요" (보안 레이어 추가)

### 실제 사례 보안 사고의 비용


 보안 사고 시나리오 

. 개발자가 실수로  키를 코드에 직접 작성
. 에 코드 업로드
. 해커가 자동화 도구로 에서  키 수집
. 해커가 회사 서버에 무단 접근
. 고객 데이터 만 건 유출

결과
- 벌금 수십억 원
- 이미지 손상 측정 불가
- 수습 비용 수억 원
- 고객 신뢰 상실 회복 불가


** 는 이런 사고를 "코드 작성 단계"에서 막아줍니다.**

## . 가지 핵심 보안 원칙

### .    (심층 방어)


 비유 성의 방어 시스템 

외곽 성벽 → 해자 → 내벽 → 감시탑 → 최후의 보루

각 층이 뚫려도 다음 층이 방어!

코드에서
- 층 방화벽
- 층 인증 (누구인가)
- 층 인가 (권한 있는가)
- 층 입력 검증
- 층 데이터 암호화


### .   (최소 권한 원칙)


 비유 호텔 열쇠 

 나쁜 예 모든 직원에게 마스터키
 좋은 예 청소부는 청소할 방만, 매니저만 금고

코드에서
- 데이터베이스 계정 읽기 전용  읽기/쓰기 분리
-  키 서비스별로 권한 최소화
- 사용자 역할 필요한 권한만 부여


### .   (보안 실패)


 비유 정전 시 도어락 

   정전 → 문 열림 (침입 가능)
   정전 → 문 잠김 (불편하지만 안전)

코드에서
- 인증 서버 장애 시 → 모든 접근 거부
- 권한 확인 실패 시 → 기본값은 "거부"
- 에러 발생 시 → 민감 정보 숨기기


### .   (입력 검증)


 비유 공항 보안 검색 

모든 승객 → 금속 탐지기 → - → 수하물 검사

코드에서
- 모든 사용자 입력  잠재적 공격
- 숫자 필드에 문자 → 거부
- 이메일 형식 불량 → 거부
-  특수문자 → 이스케이프


### .   (출력 인코딩)


 비유 외국어 번역 

한국어 "농담" → 영어로 번역 시 의미 유지

코드에서
-  출력 시 →  인코딩 ( → &&)
-  출력 시 →  인코딩
-  출력 시 →  이스케이프
- 컨텍스트에 맞는 인코딩 필수!

## . 스킬 작동 방식

### . 자동 발동 조건


파일패턴
  - "., .., .."   # 환경변수 파일
  - "**//**, **//**"          # 인증/보안 폴더
  - "*.., *.."            # 설정 파일
  - "**//**, **//**"       # 미들웨어/가드

한국어키워드
  - "보안, 시큐리티, 취약점"
  - "인증, 인가, 로그인"
  - "비밀번호, 암호화"
  - "토큰, , 세션"

영어키워드
  - ", , "
  - ", , "
  - ", , "


### . 선택적 문서 로드


 왜 선택적으로 로드하는가 

전체 문서  매우 큼 (수천 줄)
항상 전부 로드  느리고 비효율적

→ 해결 상황에 맞는 문서만 로드!

 로드 순서 

단계 언어/프레임워크 감지
  - 파일 확장자 (., ., .)
  -  문 분석
  - . 확인

단계 항상 로드 (공통)
  - /-. (시크릿 탐지)
  - -/. (체크리스트)

단계 언어별 로드
  -  → /-.
  -  → /-.
  -  → /-.

단계 상황별 로드
  -  작업 → /-.
  -  작업 → /-.
  -  보안 → /-.

## . 보안 체크리스트

### . 시크릿 관리


 필수 확인 사항 

□ 코드에  키, 비밀번호, 토큰 없음
  - 없어야 정상
  - 있으면 즉시 환경변수로 이동

□ 환경변수 또는   사용
  - 로컬 . 파일
  - 프로덕션 ,   

□ . 파일이 .에 있음
  - 없으면 에 비밀번호 노출!

□ 시크릿 로테이션 정책
  - 정기적 교체 (일마다 등)
  - 노출 시 즉시 교체


### . 입력 검증


□ 모든 사용자 입력에 검증 있음
  - 프론트엔드 검증 (용)
  - 백엔드 검증 (보안용, 필수!)

□ 에 검증 데코레이터
  - (), ()
  - (), ()
  - (), ()

□ 은   사용
  -   *    '$'
  -   *    $


### . 인증/인가


□ 모든 보호 엔드포인트에  적용
  -  제외 모든 에 인증 필요

□ 역할 기반 접근 제어() 구현
  - , ,  등 역할 분리
  - 역할별 권한 명확히 정의

□ 비밀번호는 /로 해시
  -  평문 저장
  -  ,  (취약)
  -   ( +)
  -   (최신, 권장)

## . 파일 구조


-/
├── .                      # 메인 파일 (라우터)
├── .        # 이 문서
│
├── /                         # 핵심 문서
│   ├── -.      # + 시크릿 탐지 패턴
│   └── -.         #  고급 패턴
│
├── /                    # 언어별 보안
│   ├── -.    # ., , 
│   ├── -.        # , , 
│   ├── -.            #  보안 패턴
│   └── -.          #  
│
├── /                    # 코드 템플릿
│   ├── -.            #  인증 템플릿
│   └── -.         #  보안 템플릿
│
├── /                     # 보안 패턴
│   ├── -.           #  보안 패턴
│   └── .             # 입력 검증 패턴
│
└── -/              # 빠른 참조
    └── .              # 보안 체크리스트

## . 핵심 용어 사전

 용어  설명  비유 
------------------
 **인증 ()**  누구인지 확인  신분증 확인 
 **인가 ()**  권한 확인  입장권 확인 
 **해시 ()**  단방향 암호화  믹서기 (원래대로 못 돌림)

 **시크릿 ()**  비밀 정보  금고 비밀번호

 **익스플로잇 ()**  취약점 공격  창문 깨고 침입
