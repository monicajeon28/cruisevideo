## . 왜   이 필요한가

### 비유로 이해하기 "건물 설계도"

**가 없는 개발  설계도 없이 집 짓기**

목수 "거실은 어디에"
목수 "나도 몰라, 일단 벽 쌓아"
전기공 "콘센트 어디에 넣어요"
목수 "여기"
목수 "아니, 저기"

결과 뒤죽박죽, 리모델링 비용 폭발


**   설계도 먼저!**

건축가 "여기 설계도입니다"
모두 "아, 거실은 여기, 콘센트는 저기!"

백엔드 "이  스펙대로 만들게요"
프론트엔드 "저도 이 스펙대로 호출할게요"
앱개발 "저도요!"

결과 모두가 같은 약속을 따름


### 란 무엇인가


      

비유 레스토랑 메뉴판 + 주문 시스템

메뉴판 ( 스펙)
  - "햄버거 세트" → 가격, 구성품, 옵션
  - "피자" → 가격, 사이즈, 토핑 옵션

주문 ( 호출)
  - 손님 "햄버거 세트, 콜라 라지로요" (요청)
  - 점원 "네, 잠시만요" (처리)
  - 주방 음식 준비
  - 점원 "주문하신 햄버거 세트입니다" (응답)

실제 코드에서
  - 클라이언트  /   "햄버거",  "콜라" 
  - 서버 주문 처리
  - 서버   ,  ,  "분"

## .  설계 원칙

### .  설계


      

비유 도서관 시스템

자원()  책
  - / (책 컬렉션)
  - // (특정 책)
  - /// (특정 책의 리뷰)

동작()  무엇을 할지
  -  조회 (책 읽기)
  -  생성 (새 책 등록)
  -  전체 수정 (책 정보 전체 교체)
  -  부분 수정 (가격만 수정)
  -  삭제 (책 삭제)

예시
   /          → 전체 책 목록
   //      → 번 책 상세
   /         → 새 책 등록
   //      → 번 책 전체 수정
   //    → 번 책 일부 수정
   //   → 번 책 삭제


### . 좋은  설계


좋은예시
  -  /              # 복수형
  -  //          # 로 특정
  -  ///   # 관계 표현
  -  /             # 생성
  -  //          # 수정

나쁜예시
  -  /           # 동사 포함
  -  /               # 단수형
  -  / # 쿼리스트링 남용
  -  /        # 동사 포함
  -  ///   # 으로 삭제

작성규칙
  . 명사 사용 (동사 )
  . 복수형 사용
  . 소문자 + 케밥케이스
  . 계층 관계는 / 로
  . 필터/정렬/페이징은 쿼리스트링으로
     예 /&&
