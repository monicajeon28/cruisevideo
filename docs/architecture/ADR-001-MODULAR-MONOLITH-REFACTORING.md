###   아키텍처로 점진적 리팩토링

**가 아닌 이유**
- 현재 단일 프로세스로 충분 (동영상 렌더링은 / 바운드가 아님)
- 네트워크 오버헤드 불필요 (→자막→합성 순차 처리)
- 배포 복잡도 증가 (/ 불필요)

**  선택 이유**
- 단일 프로세스 유지 → 배포 단순성
- 모듈 간 경계 명확 → 테스트 가능
- 점진적 마이그레이션 → 리스크 최소화

### 단점

#### . 마이그레이션 비용
- 예상 시간 ** // (각 주, 총 주)**

-    적용 (점진적 교체)

- 모듈  작성 (각 모듈별 역할/인터페이스 명시)
-  문서화 (설계 결정 이유 기록)

#### . 성능 오버헤드
- 모듈 간 인터페이스 호출 → 함수 호출 오버헤드

**장점**
- 완전한 독립 배포

**단점**
- 네트워크 오버헤드 (각 모듈 호출마다  왕복)
- 렌더링 시간 초 → **초** (+%)
- 배포 복잡도 증가 (,  필요)
- **과도한 엔지니어링** (현재 규모에 불필요)

**거부 이유** 렌더링 성능 저하 + 배포 복잡도 증가
